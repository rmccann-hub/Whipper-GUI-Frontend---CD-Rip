- **Re-compress now uses `-8 -e -p` (maximum effort), since the maintainer is fine spending encode time (2026-06-23, "I am always fine to add encoding time if it helps").** Reversed the earlier "stop at `-8`" call now that the only objection (encode time) is gone: `-e/--exhaustive-model-search` + `-p/--qlp-coeff-precision-search` keep `-l` at 12, so they add **no decode cost** (the dimension that matters for mobile playback) ‚Äî they cost only (much) slower *encoding* for a small extra size win (typically <1%). Validated on flac 1.4.3: `flac -8 -e -p --verify` exits clean, the file is smaller than `-5`, and the STREAMINFO decoded-PCM MD5 is **identical** (lossless confirmed). Wired via a new `_EXTRA_FLAGS = ("-e", "-p")` constant (drop to `()` to revert); per-file timeout bumped 300‚Üí600s (exhaustive search on a long track can run a while). Updated the argv test assertion, the comment, the Settings tooltip ("maximum effort‚Ä¶ slow, but encode time only"), the README equivalent-command snippet, and the CHANGELOG bullet. **LESSON:** a "compression is a bonus" decision can flip cleanly once you've separated the cost *dimensions* ‚Äî here, encode time (maintainer doesn't mind) vs decode cost (the real constraint, unchanged by `-e`/`-p`); naming the mechanism in the prior entry is what made this a one-line, well-reasoned reversal.
- **Recorded the real *why* behind "compression is a bonus": decode cost on portable players (2026-06-23, maintainer).** The maintainer supplied the rationale that high FLAC compression is bad for **mobile/low-power players** ‚Äî more CPU/battery to *decode* on playback (their ~2015-era logic, "may have changed a bit since"). Verified and made precise against the xiph preset table: the driver is the **LPC prediction order** (`-l`), which the decoder applies per sample ‚Äî whipper's default `-5` is `-l 8`, `-8` is `-l 12`, so `-8` is modestly heavier to decode. Both stay inside the FLAC **Subset** (max LPC order 12 at ‚â§48kHz), so it's a decode-*effort* difference, never a hardware-compatibility one; negligible on modern phones/PCs. Useful corollary: `-e`/`-p` keep `-l 12`, so they would add only *encode* time, not decode cost ‚Äî and whipper's `-5` is genuinely the mobile-friendly default, which re-validates keeping re-compress **off by default**. Graduated this into the `flac_recompress.py` comment, the **Settings tooltip** (user sees the tradeoff), the **README** FLAC section, and the **CHANGELOG** bullet (the off-by-default reason is now the decode cost, not just "modest gain"). Doc/comment + one tooltip string; no logic change. **LESSON:** when the maintainer hands you a decades-old rule of thumb, verify it against the canonical spec and record the *mechanism* (here: LPC order ‚Üí per-sample decode work), not just the rule ‚Äî that's what makes the default defensible later.
- **FLAC re-compress flags verified against the xiph spec + hardware test recorded (2026-06-23, maintainer: "check the flac args vs the docs; keep bit-perfect true, high compression is a bonus").** Fetched `xiph.org/flac/documentation_tools_flac.html` and checked our shipped `flac -8 --verify --silent -f -o` arg-by-arg: **all current, nothing deprecated** ‚Äî `-8` is still the max *preset* (`= -l 12 -b 4096 -m -r 6 -A "subdivide_tukey(3)"`), `--verify` is the bit-perfect guarantee (decode-in-parallel-and-compare), `--silent`/`-f`/`-o` unchanged, and FLAC-input re-encode "preserves all the metadata unless you override it" (tags/PICTURE/cuesheet/seektable) ‚Äî confirming our no-extra-flags approach. So **no flag change**; the update is recording the verification. The docs surface exactly two further-compression-but-still-lossless options, `-e/--exhaustive-model-search` and `-p/--qlp-coeff-precision-search`, both "(expensive!)" ‚Äî deliberately **not** adopted (typically <1% for several√ó the encode time; "compression is a bonus" doesn't justify it), documented as a one-line seam in `flac_recompress.py` for a future maintainer. **Recorded the hardware proof** as test-plan **Test 10**: decoded-PCM MD5 (`metaflac --show-md5sum`) identical before/after = direct bit-perfect proof independent of `--verify`; `flac --test` all-OK; PICTURE block + exported tags unchanged; files ‚â§ original (larger = red flag). The unit tests stub `flac`, so Test 10 is also the guard that a future `flac` hasn't changed the flags. Comment/doc + test-plan only, no behaviour change ‚Üí `[skip changelog]`. **LESSON:** "the args may have changed ‚Äî update" can legitimately resolve to "verified current, nothing to change"; the deliverable is the dated verification + the reasoned non-adoption of `-e`/`-p`, not a fabricated edit.
- **Optional post-rip FLAC `-8` re-compress SHIPPED (whipper-only; 2026-06-23, maintainer: "build that (opt-in, default off, with verify)").** Closed the whipper-only FLAC-compression gap (the prior entry researched it). whipper hardcodes flac `-5` (not exposable ‚Äî KDD-13); the new feature re-encodes each output FLAC at `-8 --verify` *after* the rip when the user opts in. **Design, mirroring the FLAC-verify feature it's the sibling of:** `adapters/flac_recompress.py` (`recompress_flac_files()` ‚Üí `RecompressResult`; per file runs `flac -8 --verify --silent -f -o <tmp> <orig>` then `os.replace(tmp, orig)` ‚Äî an **atomic swap-in** so a failure/crash leaves the original untouched; never raises; same "couldn't run" vs per-file-failure split as flac_verify).
