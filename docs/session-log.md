- **Black-window-during-rip is a Plasma 6 *Wayland* repaint bug → prefer XWayland (v0.3.9, 2026-06-27, BDR-209D rig).** The maintainer reported the black-out STILL happening on 0.3.8 — and the screenshot proved my 0.3.8 throttle was the wrong fix here: the progress was updating only ~once/3s (1.34→1.65→1.96%), nowhere near a flood, so the rate-limit never engaged. The real signature — the app is alive (progress ticks) but a window region covered then re-exposed stays black — is the classic Qt-on-Plasma-6-Wayland repaint failure (on Wayland the compositor re-shows the client's last-committed buffer; this Qt build mis-manages it during partial-damage updates). Fix: `app._prefer_xwayland_on_wayland()` sets `QT_QPA_PLATFORM=xcb;wayland` BEFORE QApplication on a Wayland session (XWayland repaints via X11 expose, which works) — a FALLBACK LIST so a missing xcb plugin drops back to native Wayland and can never block launch; respects an explicit user override; no-op off Wayland. Belt: a 500ms `QTimer` (`_repaint_timer`) forces a full-window `update()` while a rip runs (started/stopped with `set_rip_active`), so any stray black region self-heals even if the fallback lands on native Wayland. The real cause is a cyanrip CLI flaw: cyanrip preprocesses `-a`/`-t` with `append_missing_keys()`, which `av_strtok(src, ":")`-splits the string **naively (no `\`/quote handling)** and prepends `album=`/`album_artist=` to the first two "keyless" (`=`-less) fragments — its shorthand for `-a "Title:Artist"`. The album "Every Breath You Take: The Classics" has a colon, so our `_escape_meta_value` backslash-escapes it (`\:`) — but `append_missing_keys` ignores the `\`, sees ` The Classics` as a keyless fragment, and injects `album_artist=` before it; then `av_dict_parse_string` (which DOES honor `\:`) reads the album value straight through the injected text → `album = "Every Breath You Take:album_artist= The Classics"`. Confirmed via the cyanrip source (cyanreg/cyanrip `cyanrip_main.c`). **Conclusion: you cannot safely put a colon in a cyanrip `-a`/`-t` value via the CLI** — neither backslash nor quotes survive `append_missing_keys`'s naive split. The rip also came back `success=False`. **Fix options (pending maintainer call):** (1) feed cyanrip colon-sanitized values (`:`→`∶` U+2236, which cyanrip uses for paths anyway) so folders are clean and the parser can't choke, then authoritatively **retag the FLACs post-rip with metaflac** using the real colon (correct tags + clean folders) — the fully-correct fix, needs the cyanrip known-disc path to run the metaflac retag; (2) accept `∶` in tags too (simpler, slightly-wrong tag). **LESSON:** when feeding an external CLI a delimited mini-language, the *documented* parser (av_dict_parse_string honors `\`) may be fronted by an *undocumented* preprocessor (append_missing_keys, naive split) — escaping that satisfies one can be defeated by the other; verify against the tool's actual source, and prefer not routing user data through a fragile delimiter format at all (retag out-of-band).

- **Off-thread dep-check cross-thread bug + cancel startup-race fix (2026-06-18→20, found by post-smoke review).** Two correctness bugs, each with a verified regression test. **(1) Launch dep-check built widgets on the worker thread.** `run_dependency_check_async` connected `worker.finished` to a **lambda** capturing the per-call `gui_manager`; a signal connected to a plain callable (no QObject context) defaults to **DirectConnection**, so `_apply_dependency_report` — which builds `ManualInstallDialog`/resolver widgets — ran on the *emitting worker thread*, off the GUI thread (Qt: "QObject::setParent: ... in a different thread").
